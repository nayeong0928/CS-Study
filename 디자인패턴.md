# 디자인 패턴

소프트웨어를 설계할 때 자주 발생하는 문제들에 대한 해결 방안.

## 싱글톤 패턴

하나의 클래스에 하나의 인스턴스만 만들어지는 패턴.  

 - 장점: 자원 절약
 - 단점: 모듈간의 결합도가 높아짐 (DI로 해결 가능)

### 구현

```java

public class Singleton {

    private static class SingletonHolder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return SingletonHolder.INSTANCE;
    }
}

```

## 팩토리 패턴

구체적인 객체 생성 로직을 하위 클래스에 담는 패턴.

### WeatherFactory
날씨 타입에 따라 어떤 객체를 생성할지 결정한다.

```java

public class WeatherFactory {
    public static Weather reportWeather(WeatherType type) {
        switch (type) {
            case RAIN:
                return new Rain();
            case SUNNY:
                return new Sunny();
            default:
                throw new IllegalArgumentException("invalid weather type");
        }
    }
}

```

### 기타 클래스

날씨(Weather) 이라는 추상 클래스를 상속받는 'Rain', 'Sunny' 객체가 있다.  
날씨 타입(WeatherType)에 따라서 어떤 객체를 구현할지 결정하도록 했다.

```java

public enum WeatherType {
    RAIN, SUNNY
}

abstract class Weather {
    String report;
}

class Rain extends Weather {
    public Rain() {
        report = "오늘은 비가 옵니다.";
    }
}

class Sunny extends Weather {
    public Sunny() {
        report = "오늘은 맑습니다.";
    }
}

```

## 전략 패턴

객체의 행위를 바꾸고 싶은 경우, 직접 수정하지 않고 컨텍스트 안에서 바꿔주면서 교체가 가능하도록 한 패턴.
